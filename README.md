## What This Project Does

This is **Project 1** of a 4-part MLOps portfolio series. The goal is to demonstrate the full lifecycle of taking a machine learning model from a Jupyter notebook to a production-ready REST API:

1. **Train** a model on real customer data
2. **Wrap** it in a FastAPI web server
3. **Containerize** with Docker (runs identically everywhere)
4. **Deploy** to the cloud with a public URL

**Business problem:** A telecom company wants to identify customers likely to cancel ("churn") before they leave, so the retention team can reach out proactively.

## Architecture

```
Request (JSON)
     │
     ▼
┌─────────────┐     ┌─────────────────┐     ┌──────────────┐
│  FastAPI    │────▶│  preprocess.py  │────▶│  ML Model    │
│  app/main   │     │  (clean + scale)│     │  (.pkl file) │
└─────────────┘     └─────────────────┘     └──────────────┘
     │                                              │
     └──────────────────────────────────────────────┘
                          │
                          ▼
                  Response (JSON)
           { churn: true, probability: 0.87, risk: "High" }
```

**Data flow:**
- Raw customer data arrives as JSON
- `preprocess.py` applies the same transformations used during training
- The loaded model produces a probability score
- FastAPI returns a structured JSON response

## Dataset

**Source:** [Telco Customer Churn — Kaggle](https://www.kaggle.com/datasets/blastchar/telco-customer-churn)

- **7,043 rows** (customers)
- **21 features** (tenure, charges, contract type, add-on services, etc.)
- **Target:** `Churn` — Yes/No → 1/0
- **Churn rate:** ~26.5% (class imbalance handled with `class_weight` and `scale_pos_weight`)

## Quickstart (Local Setup)

### Prerequisites
- Python 3.11+
- Docker Desktop (for containerized run)
- Kaggle account (to download dataset)

### Step 1 — Clone and install

```bash
git clone https://github.com/YOUR-USERNAME/churn-radar.git
cd churn-radar
pip install -r requirements.txt
```

### Step 2 — Get the dataset

1. Go to [Kaggle Telco Churn dataset](https://www.kaggle.com/datasets/blastchar/telco-customer-churn)
2. Click **Download**
3. Place the CSV at: `data/raw/WA_Fn-UseC_-Telco-Customer-Churn.csv`

### Step 3 — Train the model

```bash
python src/train.py
```

This will:
- Preprocess the data
- Train Random Forest and XGBoost models
- Compare them and save the best one to `model/best_model.pkl`
- Print metrics (accuracy, F1, ROC-AUC)

Expected output:
```
Winner: XGBoost (ROC-AUC: 0.8472)
Accuracy : 81.2%
F1 Score : 0.6134
ROC-AUC  : 0.8472
```

### Step 4 — Run the API

```bash
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

Open **http://localhost:8000/docs** — you'll see the interactive API documentation.

### Step 5 — Make a prediction

```bash
curl -X POST "http://localhost:8000/predict" \
     -H "Content-Type: application/json" \
     -d '{
       "tenure": 2,
       "MonthlyCharges": 89.5,
       "TotalCharges": 179.0,
       "Contract": "Month-to-month",
       "PaymentMethod": "Electronic check",
       "InternetService": "Fiber optic"
     }'
```

Response:
```json
{
  "churn": true,
  "churn_probability": 0.8247,
  "risk_level": "High",
  "message": "This customer has a high likelihood of churning. Immediate retention action recommended."
}
```

## Run with Docker

```bash
# Build the container image
docker build -t churn-radar .

# Run the container, mapping port 8000
docker run -p 8000:8000 churn-radar

# API is now available at http://localhost:8000
```

**What's happening:**
- `docker build` reads the `Dockerfile` and creates an image
- `docker run -p 8000:8000` starts a container, forwarding port 8000 from your machine to port 8000 inside the container

## API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/` | API info and links |
| `GET` | `/health` | Health check (used by deployment platforms) |
| `POST` | `/predict` | Predict churn for one customer |
| `POST` | `/predict/batch` | Predict churn for up to 100 customers |
| `GET` | `/docs` | Interactive API documentation (Swagger UI) |

### Input Schema (`POST /predict`)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `tenure` | int | Yes | Months as a customer (0–120) |
| `MonthlyCharges` | float | Yes | Monthly bill in USD |
| `TotalCharges` | float | Yes | Total amount paid |
| `Contract` | string | Yes | `"Month-to-month"`, `"One year"`, `"Two year"` |
| `PaymentMethod` | string | Yes | Payment method |
| `InternetService` | string | Yes | `"DSL"`, `"Fiber optic"`, `"No"` |
| `OnlineSecurity` | string | No | `"Yes"` or `"No"` (default: `"No"`) |
| `TechSupport` | string | No | `"Yes"` or `"No"` (default: `"No"`) |
| `PaperlessBilling` | string | No | `"Yes"` or `"No"` (default: `"No"`) |
| `SeniorCitizen` | int | No | `0` or `1` (default: `0`) |

## Model Performance

| Model | Accuracy | F1 Score | ROC-AUC |
|-------|----------|----------|---------|
| Random Forest | ~80% | ~0.60 | ~0.84 |
| XGBoost | ~81% | ~0.61 | ~0.85 |

**Why ROC-AUC over Accuracy?**

The dataset has class imbalance (~73% No-Churn, ~27% Churn). A model that always predicts "No Churn" would have 73% accuracy but be completely useless. ROC-AUC measures the model's ability to *discriminate* between churners and non-churners regardless of the decision threshold — much more informative.

**Top churn indicators (by feature importance):**
1. `Contract` type — month-to-month customers churn most
2. `tenure` — newer customers churn more than long-term ones
3. `TotalCharges` — correlated with tenure
4. `MonthlyCharges` — higher bills → higher churn
5. `InternetService` — fiber optic customers churn more

## Deploy to Render (Free)

1. Push this repo to GitHub
2. Go to [render.com](https://render.com) → **New** → **Web Service**
3. Connect your GitHub repo
4. Settings:
   - **Environment:** Docker
   - **Port:** 8000
5. Click **Deploy**

Render reads your `Dockerfile` and deploys automatically. You'll get a public URL like `https://churn-radar-xxxx.onrender.com`.
